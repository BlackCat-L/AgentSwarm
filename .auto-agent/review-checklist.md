# 代码审查清单

> AI 完成每个任务的 Implement 阶段后，对照此清单自检。
> 部署到目标项目后，根据项目技术栈补充具体检查项。

## 使用前

- `estimated_complexity = low` → 只执行「架构合规性」+「Sprint Contract 验收」两节，跳过评分表
- `estimated_complexity = medium / high` → 执行全部清单 + 评分表

---

## 架构合规性

<!-- 根据项目类型填写，示例： -->

- [ ] 新模块遵循项目分层架构（如 CLI 层 / 业务层 / 数据层）
- [ ] 新功能在正确的层级实现，没有跨层污染
- [ ] 没有绕过已有抽象直接调用底层实现
- [ ] 新增模块已在架构入口注册（如 `CLAUDE.md` 模块列表、DI 容器等）

## 代码风格

- [ ] 命名规范与项目现有代码一致
- [ ] 文件组织遵循项目目录结构约定
- [ ] 注释使用项目统一格式
- [ ] 没有 TODO/FIXME 遗留（或已有跟踪 ticket）

## 可复用性

- [ ] 没有重复实现已有功能（先 Grep 搜索相似逻辑）
- [ ] 优先复用现有模块和工具函数
- [ ] 三个相似逻辑已抽象为公共函数（两个以下保留观察）
- [ ] 新引入的依赖是项目已有或经过审查的

## 健壮性

- [ ] 空值/边界情况有处理（null check、空数组、零值）
- [ ] 异常处理覆盖了可预见的失败路径
- [ ] 错误信息包含足够的调试上下文
- [ ] 外部输入有验证（用户输入、API 响应、文件读取）

## 安全性

- [ ] 敏感数据（密钥、token、密码）没有硬编码
- [ ] 环境变量/配置文件中的敏感信息已 gitignore
- [ ] 没有命令注入风险（用户输入不拼接到 shell 命令）
- [ ] 没有 SQL 拼接（使用参数化查询）

## 性能

- [ ] 没有在热路径中进行不必要的 I/O 或网络请求
- [ ] 循环/递归有合理的退出条件
- [ ] 大文件/大数据有分页或流式处理
- [ ] 资源使用后有释放/关闭（文件句柄、连接池、数据库连接）

## Sprint Contract 验收

- [ ] 所有 `acceptance_criteria` 已满足
- [ ] 涉及文件与 `critical_files` 列表一致
- [ ] 没有实现契约之外的功能（防止范围蔓延）
- [ ] 没有遗漏契约要求的功能

---

## Evaluator 评分（1-5 分）

| 维度 | 权重 | 评分 | 说明 |
|------|------|------|------|
| **功能正确性** | 40% | _/5 | 是否实现 Sprint Contract 要求的所有功能 |
| **架构合规** | 25% | _/5 | 是否符合项目架构规范 |
| **代码质量** | 20% | _/5 | 可读性、健壮性、安全性、异常处理 |
| **复用性** | 15% | _/5 | 是否复用现有模式，无重复造轮子 |

**通过条件**: **所有维度均达到各自的 hard threshold**，且总分 ≥ 3.5

| 维度 | Hard Threshold |
|------|---------------|
| 功能正确性 | ≥ 4 |
| 架构合规 | ≥ 3 |
| 代码质量 | ≥ 3 |
| 复用性 | ≥ 3 |

**评分参考（Few-shot 校准）**:
- **5 分**: 超出预期，代码可直接合入主干，无需修改
- **4 分**: 满足契约，有小问题但不影响功能（如命名、注释）
- **3 分**: 基本满足，有明显问题需修改（如缺少异常处理、硬编码）
- **2 分**: 功能不完整，或严重违反架构规范
- **1 分**: 功能错误，无法编译/运行，或引入严重 bug
